## Plan 2 — 프로덕션 hotfix 2개 (검증 중 발견)

사용자가 프로덕션에서 OAuth 시도 → 두 가지 실제 환경 문제 노출. spec에는 잡히지 않았던 카페24 API/Supabase 인프라 사실:

### `74845df` — cafe24 validate endpoint 변경

`/admin/store`는 우리 5 scope에 없는 권한 요구 → 토큰은 valid한데 403 → "scope 권한이 부족합니다" 표시. `/admin/categories?limit=1`(mall.read_category)로 변경.

### `81d4102` — vault public schema wrapper

supabase-js `.schema('vault').rpc('create_secret', ...)`가 "Invalid schema: vault"로 throw. **PostgREST가 기본적으로 `vault` schema를 expose 안 함**. 해결책으로 `public.create_vault_secret(secret, name, description)` + `public.delete_vault_secret(secret_id)` SECURITY DEFINER wrapper 함수를 만들고 코드가 그쪽을 호출하게 변경.

## 워크플로 메모

- Plan 1은 inline로 controller 직접 실행 (단순). Plan 2는 subagent-driven으로 분할 — fresh subagent per task + review로 컨텍스트 오염 최소화. 중간 build 에러나 nested-button defect 같은 패턴 문제는 fix subagent로 빠르게 회수.
- spec → plan 단계에서 해결된 vault `.schema('vault')` 가정이 실제 환경에서 작동 안 함 → 프로덕션 hotfix로 발견. **plan은 가설, hotfix가 진짜 그라운드 트루스**. 다음 plan부터는 외부 서비스 sandbox로 사전 검증 단계를 별도 task로 둘 가치 있음.
- 카페24 admin/store endpoint가 별도 scope 요구하는 사실도 같은 맥락 — 코드 작성 단계에선 안 잡힘. validate endpoint는 실제 우리 scope에 있는 권한 기반으로 골라야 함.
- 사용자 검증 단계에서 환경변수 등록 + 캐시 무효화 재배포가 핵심 트랩. 함수가 옛 env로 실행되면 디버깅 시간 낭비. 다음 plan부터는 "환경변수 추가 후 캐시 해제 재배포 트리거" 한 줄을 모든 환경변수 task의 마지막 step에 명시.
